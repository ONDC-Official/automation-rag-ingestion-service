## Purpose of This Document Set

This documentation corpus defines and explains the **official test cases and validation rules** for the ONDC `ONDC:FIS12` protocol under the domains and version listed above.

It serves two audiences:

1. **Protocol participants** (BAPs, BPPs) who need to understand exactly which payload fields are required, what values are valid, and which requests will be synchronously NACKed by the network.
2. **Code generators and RAG systems** that need a machine-readable understanding of the JVAL DSL, the validation rule structure, and how to navigate the documentation to answer questions about specific actions or fields.

> **Key term — NACK:** A synchronous rejection response sent immediately when an incoming API request fails one or more required validations before any business logic executes. All validations defined in the x-validations config are treated as NACK-producing rules.

## Document Map

The following files make up this knowledge base. Each description explains what questions a particular file can answer.

### `context.md` _(this file)_

- What protocol, domain, and version these docs cover.
- How all the other documents relate to each other.
- Terminology glossary and navigation guide.

### `x-validation_details.md`

- **Primary reference for the JVAL DSL** (the language used to write validation rules).
- Covers the full config structure (`_TESTS_`, `_SESSION_DATA_`), every field in a test object (`_NAME_`, `_SCOPE_`, `_CONTINUE_`, `_RETURN_`, `_ERROR_CODE_`, etc.), variable binding via JSONPath, scoping, conditional skip logic, nested tests, and session/cross-request state.
- **Go here to understand how a validation rule is written or what any field name means.**

### `validations/<api>.md`

- **Human-readable descriptions for every named validation test**, organised by API action (`search`, `on_search`, `select`, `on_select`, `init`, `on_init`, `confirm`, `on_confirm`, `cancel`, `on_cancel`, `track`, `on_track`, `update`, `on_update`, `status`, `on_status`).
- Each entry corresponds to a test name (e.g. `REQUIRED_CONTEXT_DOMAIN`) and describes in plain language what the test checks, what must be present or matched, and any skip conditions.
- **Go here to understand what a specific test name means, or to verify that an x-validation rule is correct.**

### `api schemas/<action>.md`

- **Per-action schema trees** generated from the canonical `validPaths.json` — one file per API action (e.g. `api schemas/search.md`, `api schemas/on_confirm.md`).
- Each file renders the valid JSONPath field paths for that action as a nested, typed schema with reusable type definitions extracted and cross-referenced via `$ref:TypeName`.
- Includes example full JSONPath expansions at the bottom of each file.
- **Go here to check whether a JSONPath is legal for a given action, to browse available fields, or to understand the payload shape.**

### `x-validations/<action>.md`

- **The actual machine-readable x-validation config** for this domain/version, split into one file per API action (e.g. `x-validations/search.md`, `x-validations/on_confirm.md`).
- Each file contains the full `_TESTS_` and `_SESSION_DATA_` objects for that action in their compiled JVAL format, with YAML frontmatter (action, domain, version, numTests).
- This is the source of truth for which rules are active; `validations/<api>.md` is the human-readable companion to it.
- **Go here to inspect or reason about the exact structure of a live validation rule, or to write new rules following the same pattern.**

### `dsl/README.md`

- **Entry-point overview of the JVAL Return Expression DSL** — the mini-language used inside the `_RETURN_` field of every test object.
- Explains the end-to-end pipeline (string → tokens → CST → AST → code/markdown), gives a worked example, and maps to the sub-documents.
- **Go here first when questions are about the `_RETURN_` expression language itself.**

### `dsl/grammar.md`

- **Formal EBNF grammar** for the JVAL expression language.
- Covers operator precedence rules: `!` (NOT) > `&&` (AND) > `||` (OR).
- Lists every terminal token: binary functions (`all in`, `any in`, `none in`, `follow regex`, `equal to`, `greater than`, `less than`), unary functions (`are unique`, `are present`), and Boolean connectives.
- **Go here for precise syntax rules or to understand how a complex expression is parsed.**

### `dsl/operations.md`

- **Semantic reference for every JVAL operator and built-in function** — what each one does at runtime, usage examples, and how `CompileToMarkdown` renders it as documentation.
- **Go here to understand what `all in`, `follow regex`, `are present`, etc. mean in practice.**

### `dsl/ast.md`

- **AST node type reference** — describes every node shape (`ReturnStatementNode`, `BinaryOperatorNode`, `NotOperatorNode`, `CustomBinaryFunction`, `CustomUniaryFunction`, `IdentifierNode`).
- **Go here when working with the parsed AST or writing a code generator that consumes AST nodes.**

### `dsl/pipeline.md`

- **Full compiler pipeline diagram** from `_RETURN_` string to runnable validation code or human-readable markdown.
- Steps: Lexer (`tokens.ts`) → Parser (`parser.ts`) → AST Builder (`ast.ts`) → Code Generator or Documentation Generator.
- Public entry point: `buildAstFromInput(string)` from `combined.ts`.
- **Go here to understand how the compilation works end-to-end, or to find the right source file for each stage.**

## Key Concepts Glossary

| Term                  | Definition                                                                                                                                                                         |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ONDC**              | Open Network for Digital Commerce — an open protocol network for e-commerce in India.                                                                                              |
| **BAP**               | Buyer App Platform — initiates API calls (e.g. `search`, `init`, `confirm`).                                                                                                       |
| **BPP**               | Seller App Platform — responds to API calls (e.g. `on_search`, `on_confirm`).                                                                   |
| **x-validation**      | A single named validation rule expressed in JVAL. Lives inside `_TESTS_` keyed by action name.                                                                                     |
| **JVAL**              | JSON Validation Language — the declarative DSL used in this project to write validation rules as JSON/YAML configs.                                                                |
| **`_TESTS_`**         | Top-level key in the x-validations config. Object keyed by action name; each value is an ordered array of test objects.                                                            |
| **`_SESSION_DATA_`**  | Top-level key in the x-validations config. Declares JSONPath expressions whose values are extracted and saved after each action for use in later cross-request validations.        |
| **Test object**       | A single validation rule — a JSON object with system fields (`_NAME_`, `_SCOPE_`, `_CONTINUE_`, `_RETURN_`, etc.) and user-defined variable bindings.                              |
| **`_RETURN_`**        | The field in a test object that holds a JVAL expression string (or an array of nested test objects). This expression is the actual pass/fail condition.                            |
| **`_CONTINUE_`**      | Conditional skip — if this JVAL expression evaluates to false, the test is skipped (not failed). Used to express "skip if field not present".                                      |
| **`_SCOPE_`**         | A JSONPath that narrows the root document for all variable lookups in this test object.                                                                                            |
| **`_ERROR_CODE_`**    | The error code emitted when the test fails.                                                                                                                                        |
| **Variable binding**  | A user-defined field in a test object whose value is a JSONPath string or a literal array. The resolved value is available to `_RETURN_` by the field's name.                      |
| **JSONPath**          | A query language for JSON, like XPath for XML. Used in variable bindings and `_SESSION_DATA_` to extract values from payloads. Must resolve to a path listed in the corresponding `api schemas/<action>.md`. |
| **NACKed (sync)**     | A request that is rejected synchronously — the response is a NACK (negative acknowledgement) before any business logic runs. All test failures cause a NACK.                       |
| **`_EXTERNAL`**       | A reserved JSONPath prefix (`$._EXTERNAL`) giving access to session data saved from previous API calls. Used for cross-request/stateful validation.                                |
| **CompileToMarkdown** | The pipeline stage that converts a JVAL AST into human-readable bullet-point documentation (the format seen in `outputs.md`).                                                      |
| **Chevrotain**        | The parser toolkit used to implement the JVAL lexer and parser (`tokens.ts`, `parser.ts`).                                                                                         |
| **AST**               | Abstract Syntax Tree — the language-agnostic typed tree produced by the JVAL compiler's AST builder, consumed by code generators.                                                  |

## ONDC API Actions

The validations in this corpus are organised by ONDC API action. Common actions in scope:

| Direction     | Actions                                                                                              |
| ------------- | ---------------------------------------------------------------------------------------------------- |
| **BAP → BPP** | `search`, `select`, `init`, `confirm`, `cancel`, `track`, `update`, `status`                         |
| **BPP → BAP** | `on_search`, `on_select`, `on_init`, `on_confirm`, `on_cancel`, `on_track`, `on_update`, `on_status` |

Every test object belongs to exactly one action. The test name uniquely identifies it within that action's test set.

## How the System Works End-to-End

```
x-validations.json  (JVAL config)
        │
        │  _TESTS_["search"][i]._RETURN_ = "domain all in validDomains && action are present"
        │
        ▼
JVAL Compiler (dsl/pipeline.md)
  Lexer → Parser (CST) → AST Builder → Code Generator
        │
        ▼
Generated validation function (TypeScript / Python / JS / Go)
        │
        ▼
Runtime: receives API payload, runs all tests for the action
        │
        ├── PASS → proceed
        └── FAIL → NACK response + error code from _ERROR_CODE_
                   description from outputs.md shown in test report
```
